# 红黄绿灯模型

用红黄绿灯快速判断点子是否值得继续。这是最基础的淘汰工具——
如果红灯亮了，不需要继续评估，直接给出放弃建议。

## 核心问题

对用户的点子，从以下 5 个维度各打一盏灯：

1. **必要性**：没有这个产品，目标用户的问题有多痛？
   - 🟢 必须解决（痛点明确、现有方案差）
   - 🟡 有了更好（有替代方案但不够好）
   - 🔴 锦上添花（问题不痛或已有好方案）

2. **差异化**：和现有方案相比，有什么本质不同？
   - 🟢 明显差异化（10x 更好或全新方式）
   - 🟡 有一些不同（更好用/更便宜/更垂直）
   - 🔴 没有差异（me-too 产品）

3. **可触达**：你能接触到目标用户吗？
   - 🟢 清晰的渠道（你已经在用户群体中）
   - 🟡 需要建立渠道（有思路但需要验证）
   - 🔴 不知道怎么找到用户

4. **付费意愿**：用户愿意为此付钱吗？
   - 🟢 已有付费习惯（同类产品用户在付费）
   - 🟡 可能付费（需要教育市场）
   - 🔴 免费替代品太多/用户不愿付费

5. **OPC 可行性**：一个人能做吗？（综合考虑技术、运营、合规三个维度）
   - 🟢 技术简单、运营轻量、无特殊合规要求，一个人能搞定
   - 🟡 需要学习新技能或有一些运营负担，但总体可控
   - 🔴 技术复杂 / 需要持续人工运营（如 24h 审核）/ 有合规门槛（牌照/备案），
     一个人做不了或风险太高

   **注意**：OPC 可行性不只是"技术能不能实现"。很多时候技术简单但运营复杂
   （比如匿名社交、社区产品需要持续审核内容），或者技术简单但合规门槛高
   （比如涉及金融、医疗需要资质）。评估时三个维度都要看：
   - 技术可行性：一个人能写出来吗？
   - 运营可行性：一个人能维护得动吗？（审核、客服、社区管理）
   - 合规可行性：一个人能搞定合规要求吗？（备案、资质、法律风险）

## 判定规则

```
任意维度 🔴 → 整体结论：建议放弃（或大幅修改点子）
3+ 个维度 🟢 → 整体结论：值得认真评估（进入下一个模型）
其余情况 → 整体结论：需要更多信息（进入致命假设验证）
```

**重要：红灯意味着"停"。不要试图用其他维度的绿灯来抵消红灯。
如果用户的问题不痛（必要性红灯），再好的技术也没用。**

## OPC 视角适配

OPC 的资源和精力有限，以下情况需要格外谨慎：

- 如果"OPC 可行性"是红灯，这不是"克服困难"的问题，是"选错赛道"的问题。
  特别是运营复杂度——如果产品上线后需要你每天花大量时间审核内容、处理纠纷、
  维护社区氛围，那你不是在做产品，是在做全职运营。
- 如果"付费意愿"是黄灯，OPC 没有资源做市场教育，这是大公司的活
- 必要性 + 差异化必须至少有一个是绿灯，否则连入场资格都没有
- **运营复杂度红灯 = 技术红灯**：即使技术很简单，如果运营一个人扛不住，
  对 OPC 来说结果一样——做不了

## 团队视角适配

将"OPC 可行性"维度替换为"团队可行性"，评判标准不同：

- **团队可行性**：团队现有的人力和技能能覆盖吗？
  - 🟢 团队有匹配的人才，带宽充裕
  - 🟡 需要招人或学习，但可控
  - 🔴 团队缺乏关键技能且短期无法补齐

团队特有的考量：
- 如果团队 3+ 人，可以承担比 OPC 更复杂的项目，"大项目"象限不一定是红灯
- 但团队意味着更高的运营成本，"付费意愿"黄灯同样是风险——你需要更多收入覆盖团队成本
- 必要性 + 差异化仍必须至少有一个绿灯，这个规则不因团队而改变

```markdown

| 维度 | 灯 | 判断依据 |
|------|----|----------|
| 必要性 | 🟢/🟡/🔴 | ... |
| 差异化 | 🟢/🟡/🔴 | ... |
| 可触达 | 🟢/🟡/🔴 | ... |
| 付费意愿 | 🟢/🟡/🔴 | ... |
| {OPC: OPC 可行性 / 团队: 团队可行性} | 🟢/🟡/🔴 | ... |

**整体判断：🟢 值得推进 / 🟡 需要验证 / 🔴 建议放弃**
```
