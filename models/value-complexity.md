# 价值-复杂度矩阵

用二维坐标判断点子的投入产出比。把点子放在"价值-复杂度"坐标系里，
快速看出哪些值得做、哪些应该放弃。

## 核心问题

对用户的点子评估两个维度：

### 价值（纵轴）：用户获得的价值有多大？
- **高价值**：解决核心痛点、节省大量时间/金钱、创造新能力
- **中价值**：改善体验、提高效率、降低成本
- **低价值**：锦上添花、边缘功能、nice-to-have

### 复杂度（横轴）：实现起来有多难？
考虑个人/团队实际能做到的程度：
- **低复杂度**：一个周末能出 MVP，技术栈熟悉，无外部依赖
- **中复杂度**：1-4 周能出 MVP，需要学一些新东西，有一些外部依赖
- **高复杂度**：需要 1 个月以上，涉及复杂技术/合规/多方协作

## 四象限判定

```
         高价值
           |
    快速赢  |  大项目
   (先做这个)| (OPC 慎重)
           |
低复杂度 -------- 高复杂度
           |
   没意义  |  填充物
  (直接放弃)| (别浪费时间)
           |
         低价值
```

| 象限 | 判定 | 建议 |
|------|------|------|
| 高价值 + 低复杂度 | 🟢 快速赢 | 立即开始，这是最好的机会 |
| 高价值 + 高复杂度 | 🟡 大项目 | OPC 慎重；团队可以做但要拆分找到最小切入口 |
| 低价值 + 低复杂度 | 🟡 填充物 | 可以做但不值得投入太多精力 |
| 低价值 + 高复杂度 | 🔴 没意义 | 直接放弃，没有任何理由值得做 |

## OPC 视角适配

OPC 的核心竞争力是速度和灵活性。对 OPC 来说：

- **复杂度的计算方式和团队不同**：对团队来说"高复杂度"可能只是"需要 3 个人一个月"，
  对 OPC 来说是"我需要学一门新技术 + 对接 3 个外部系统 + 还要做客服"——这实际上是不可行的

- **OPC 应该追求高价值 + 低复杂度的象限**：这是 Pieter Levels 的核心策略——
  快速出产品、快速验证、快速迭代。大项目留给有团队的玩家。

- **如果点子落在"大项目"象限**：不要直接放弃，而是问"能不能找到一个子集，
  把它拉到'快速赢'象限？"如果找不到，再放弃。

## 团队视角适配

团队可以承担更高的复杂度，但代价也不同：

- **复杂度阈值更高**：团队 3 人可以做 OPC 一个人做不了的事。"高复杂度"对团队可能只是
  "需要 1-2 个月"，而不是"不可能"。但要注意协调成本——3 个人的沟通开销是真实存在的。

- **"大项目"象限不一定是坏事**：团队适合做有价值的大项目。关键是确认团队有持续投入的
  耐心和资源。问自己："团队愿意在这个项目上投入 3-6 个月吗？"

- **"填充物"象限更要警惕**：低价值 + 低复杂度对团队来说更危险——做了不值得，
  因为团队的时间成本远高于 OPC。

- **收入要求更高**：团队需要覆盖多人的成本。同样一个"中价值"的点子，
  OPC 可能觉得够生活，团队可能入不敷出。

```markdown
## 价值-复杂度矩阵

**坐标位置：** 高/中/低价值 × 低/中/高复杂度
**所在象限：** 快速赢 / 大项目 / 填充物 / 没意义
**判定：** 🟢/🟡/🔴

### 价值评估
- 用户价值：...
- 付费意愿匹配：...

### 复杂度评估
- 技术复杂度：...
- 时间估算：...
- 外部依赖：...

### 如果在"大项目"象限，可能的切入口
- 缩小到：...
- 价值降级换取复杂度降低是否可接受：...
```
